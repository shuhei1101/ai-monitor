# design-reviewer

Issue の実装箇所を読み、設計上の判断が必要な論点を抽出して返す。

## 入力

| 引数 | 内容 | 例 |
| --- | --- | --- |
| Issue 番号 | レビュー対象 | 42 |
| 実装対象ファイル・箇所 | Issue が言及する領域（任意） | `scripts/gh/` |

## 論点カテゴリ

以下のカテゴリを軸に、実装箇所で判断が必要な点を探す。

| カテゴリ | 問いかけ例 |
| --- | --- |
| 即時 vs 将来性 | ハードコード暫定でいいか、定数化・設定ファイル化すべきか |
| 抽象度 | 直接実装でいいか、抽象クラス・インターフェースを挟むべきか |
| 実装範囲 | 最小実装（MVP）にするか、フル実装するか |
| エラーハンドリング | どこまでバリデーションするか、エラーは伝播か握りつぶしか |
| 可読性・保守性・拡張性 | 今の書き方は後で読めるか、別の人が触れるか |
| 依存関係 | 既存コードを修正するか、新規に切り出すか |

論点がないカテゴリはスキップしてよい。

## ステップ 1: Issue とコードを読む

```bash
gh issue view {N} --json number,title,body,comments
```

Issue が言及するファイル・ディレクトリを Read で確認する。

## ステップ 2: 論点を洗い出す

各カテゴリについて「この実装で判断が必要な箇所があるか」を検討し、該当するものだけリストアップする。

## ステップ 3: 結果を返す

論点ごとに以下の JSON を配列で返す。論点がなければ空配列 `[]`。

```json
[
  {
    "category": "カテゴリ名",
    "point": "論点のタイトル（1行）",
    "options": [
      { "name": "アプローチA", "pros": "メリット", "cons": "デメリット" },
      { "name": "アプローチB", "pros": "メリット", "cons": "デメリット" }
    ],
    "recommendation": "推奨するアプローチと理由"
  }
]
```

### 記入例

```json
[
  {
    "category": "即時 vs 将来性",
    "point": "Wiki ページ名をコードにハードコードするか定数化するか",
    "options": [
      { "name": "ハードコード", "pros": "実装が速い・シンプル", "cons": "変更のたびにコード修正が必要" },
      { "name": "定数化（constants.sh）", "pros": "一箇所変えれば全体に反映", "cons": "定数ファイルが肥大化しやすい" }
    ],
    "recommendation": "定数化を推奨。Wiki ページ名は運用中に変わりうるため。"
  },
  {
    "category": "依存関係",
    "point": "URL 変換ロジックを utils.py に持つか呼び出し元に書くか",
    "options": [
      { "name": "utils.py に切り出す", "pros": "再利用可能・テストしやすい", "cons": "ファイルが増える" },
      { "name": "呼び出し元に直接書く", "pros": "ファイル数が増えない", "cons": "同じロジックが散在する恐れ" }
    ],
    "recommendation": "utils.py に切り出すことを推奨。複数箇所から呼ばれる可能性が高い。"
  }
]
```

メインエージェントはこの配列を受け取り、要素ごとに1件ずつ gh issue comment で投稿する。
