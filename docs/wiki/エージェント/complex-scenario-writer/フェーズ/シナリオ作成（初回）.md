# シナリオ作成（初回）

親 epic Issue の `## ユースケース一覧` から複合ユースケースシナリオを設計し、epic ブランチに commit + 索引更新 push した上で待機に入る。

## 手順

### 親 epic Issue の確認

epic PR 本文の `## 紐づく Issue` から親 epic Issue 番号を得て MCP `get_issue_or_pr` を呼ぶ:
- `number`: 親 epic Issue 番号
- `is_pr`: false
- `comments`: false

epic Issue 本文の `## ユースケース一覧` の各 UC 名（`単一ユースケース/{UC名}.md` のノード名になる）と `## 横断要件` を抽出する。
画面変更ありの epic では epic PR 本文の `## UI 設計` も参照する。

### シナリオファイルの新規作成

テンプレート（シナリオ）に従い、`docs/wiki/設計図/シナリオ/複合ユースケース/{機能名}.md` を新規作成する（`{機能名}` は epic Issue のタイトルまたはドメイン名から決める）。

- 中間ノードは `{UC名}:{シナリオ見出し}` 形式で `## ユースケース一覧` の UC 名と 1:1 対応
- 検証したい UC 間のつなぎ目を `subgraph FOCUS["検証対象: {要約}"]` で囲む
- 各 UC ノードに `click {ノードID} "../単一ユースケース/{UC名}.md#{アンカー}"` を並べる
- 正常シナリオ + 起こりうる異常シナリオを H2 で並列に書く

### 索引の更新

`docs/wiki/設計図/シナリオ/README.md` の索引に行を追加する（種別カラム = `複合ユースケース`）。

### commit + push

epic ブランチの worktree（`.claude/worktrees/{ブランチ名}` = `${BRANCH}` を `/` → `-` 置換したパス）で以下を実行する:

- `git add docs/wiki/設計図/シナリオ/複合ユースケース/{機能名}.md docs/wiki/設計図/シナリオ/README.md`
- `git commit -m "{機能名}複合シナリオを追加"`
- `git push`

### 完了報告の投稿

MCP `comment` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true
- `sender`: `complex-scenario-writer`
- `receiver`: ユーザーログイン名（`gh api user --jq '.login'` で取得）
- `body`: シナリオ作成の完了報告（シナリオファイルへのリンク + 正常 / 異常シナリオの要点 + 承認・修正方法の案内）

### 議論中 付与 + 待機

MCP `add_labels` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true
- `labels`:
  - `$AI_MONITOR_LABEL_IN_DISCUSSION` の値

続けて MCP `set_assignee` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true

### 作業完了報告

MCP `report_completion` を呼ぶ:
- `agent_name`: `complex-scenario-writer`
- `number`: $pr_number
