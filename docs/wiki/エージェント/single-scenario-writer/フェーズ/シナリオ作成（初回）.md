# シナリオ作成（初回）

親 story Issue の `## ユースケース要件` から単一ユースケースシナリオを設計し、story ブランチに commit + 索引更新 push した上で待機に入る。

## 手順

### 親 story Issue の確認

story PR 本文の `## 紐づく Issue` から親 story Issue 番号を得て MCP `get_issue_or_pr` を呼ぶ:
- `number`: 親 story Issue 番号
- `is_pr`: false
- `comments`: false

story Issue 本文の `## ユースケース要件` と `## 背景` を読み、対象 UC の名前（story タイトル = 親 epic の `## ユースケース一覧` の UC 名 = 単一UC ファイル名）と要件・関連する epic 横断要件を把握する。

### シナリオファイルの新規作成

テンプレート（シナリオ）に従い、`docs/wiki/設計図/シナリオ/単一ユースケース/{UC名}.md` を新規作成する（`{UC名}` は story タイトル）。

- 冒頭リードに対応エージェント（該当する場合）と対応テストファイル行のプレースホルダを置く
- 正常シナリオ + 起こりうる異常シナリオを H2 で並列に書く
- 各シナリオは `### セットアップ` / `### フロー` / `### 期待値` の 3 サブセクション構成
- フロー図の粒度は「ユーザーの 1 操作 → サブシステム貫通」。1 UC の内部で結合を跨いだ流れを sequenceDiagram で描く

### 索引の更新

`docs/wiki/設計図/シナリオ/README.md` の索引に行を追加する（種別カラム = `単一ユースケース`）。

### commit + push

story ブランチの worktree（`.claude/worktrees/{ブランチ名}` = `${BRANCH}` を `/` → `-` 置換したパス）で以下を実行する:

- `git add docs/wiki/設計図/シナリオ/単一ユースケース/{UC名}.md docs/wiki/設計図/シナリオ/README.md`
- `git commit -m "{UC名}単一シナリオを追加"`
- `git push`

### 完了報告の投稿

MCP `comment` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true
- `sender`: `single-scenario-writer`
- `receiver`: ユーザーログイン名（`gh api user --jq '.login'` で取得）
- `body`: シナリオ作成の完了報告（シナリオファイルへのリンク + 正常 / 異常シナリオの要点 + 承認・修正方法の案内）

### 議論中 付与 + 待機

MCP `add_labels` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true
- `labels`:
  - `$AI_MONITOR_LABEL_IN_DISCUSSION` の値

続けて MCP `set_assignee` を呼ぶ:
- `number`: $pr_number
- `is_pr`: true

### 作業完了報告

MCP `report_completion` を呼ぶ:
- `agent_name`: `single-scenario-writer`
- `number`: $pr_number
