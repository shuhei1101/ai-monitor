# 要件確定（初回）

story Issue 本文（前提条件 / 概要 / 背景 / ユースケース要件）の草案を作成し、確認事項を投稿して待機に入る。

## 手順

### 親 epic の確認

MCP `get_issue_or_pr` を呼び、親 epic の内容を読む:
- `number`: 初期処理で取得した `parent` の番号
- `is_pr`: false
- `comments`: false

親 epic の `## ユースケース一覧` の UC 名と story Issue のタイトルを照合し、自 story が担当する UC を特定する。あわせて `## 横断要件` を把握する。

### 本文の作成

テンプレート（イシュー本文/ストーリー）に沿って 4 セクション（`## 前提条件` / `## 概要` / `## 背景` / `## ユースケース要件`）の草案を作成する。

- `## 背景` に「親 epic #{番号} の UC「{UC 名}」に対応」の 1 行を必ず含める
- `## ユースケース要件` に epic 横断要件を参照する行がある場合、補足に「epic 横断要件「{要件の要旨}」に基づく」を明記する

MCP `update_body` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `body`: 作成した本文

### 完了報告の投稿

MCP `comment` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `sender`: `story-conductor`
- `receiver`: ユーザーログイン名（`gh api user --jq '.login'` で取得）
- `body`: 本文草案の完了報告（確認してほしい点の要約 + 承認・修正方法の案内）

### 確認質問の投稿

以下の観点から今回の story に該当するものだけを抽出し、本文・親 epic の内容から AI 側で推奨案を判断して投稿する。
抽出結果が 0 件のときは `ask_questions` を呼ばずに次の手順へ進む。

| 観点 | 質問 | 選択肢例 |
| --- | --- | --- |
| UC 粒度 | 「この story は 1 UC で表現できていますか？」 | A. 1 UC で表現できている / B. 複数 UC に分けたほうがよい（{分割案}） |
| スコープ境界 | 「{周辺機能} も UC に含めますか？」 | A. 含めない（今回はスコープ外） / B. 含める |
| 横断要件の適用 | 「epic 横断要件「{要件}」は本 UC でも適用しますか？」 | A. 適用する / B. 適用しない（{理由}） |

MCP `ask_questions` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `sender`: `story-conductor`
- `receiver`: ユーザーログイン名
- `intro`: `以下の点についてご確認ください。記号（例: A）で返信いただければ、その方針で進めます。`
- `questions`: 上記観点表から抽出

### 議論中 付与 + 待機

MCP `add_labels` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false
- `labels`:
  - `$AI_MONITOR_LABEL_IN_DISCUSSION` の値

続けて MCP `set_assignee` を呼ぶ:
- `number`: $issue_number
- `is_pr`: false

### 作業完了報告

MCP `report_completion` を呼ぶ:
- `agent_name`: `story-conductor`
- `number`: $issue_number
